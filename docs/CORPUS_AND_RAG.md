# Корпус прошлых тендеров и векторный контекст

## Что может сделать этот репозиторий сейчас

**Прямого доступа к вашему Google Drive из Cursor/CI нет** — документы лежат у вас под учётной записью сервисного аккаунта и не «просматриваются» ассистентом без вашего локального запуска команд.

Чтобы **исследовать накопленную базу** поданных тендеров в структуре `_lena/tenders/…`, запустите у себя:

```bash
node src/cli.js drive tenders-inventory "<url_или_id_корневой_папки>"
```

### Архив «тендеры 2025» (фиксированная папка, сотни часов работы)

Папка года **не обязана** совпадать с `_lena/tenders/2025`: это может быть **отдельный архив** на Диске. **Лена в текущем коде не «читает» и не запоминает** содержимое всех PDF/DOC автоматически: для этого нужен отдельный пайплайн (извлечение текста → обезличивание → курирование или векторный индекс). См. раздел [Векторная база (RAG)](#векторная-база-rag).

Что уже можно сделать **одной командой** — получить **полный рекурсивный манифест** всех файлов под корнем архива (только метаданные: путь, тип, ссылки), с лимитами глубины и числа файлов:

```bash
node src/cli.js drive corpus-manifest "https://drive.google.com/drive/folders/ВАШ_АРХИВ_2025" 40 200000
```

Аргументы: `[maxDepth]` (по умолчанию 30), `[maxFiles]` (по умолчанию 100000). Если `capped: true` — лимит достигнут, обход можно повторить с большим `maxFiles` или разбить архив по подпапкам.

Дальше манифест используют **люди или внешний сервис**: выбор файлов для выжимок в `_lena/context`, или загрузка в parserit/RAG.

**Формат ответа `corpus-manifest`:** JSON с полями `rootFolderId`, `fileCount`, `folderCount`, `capped`, `files[]` (у каждого файла: `path` относительно корня архива, `mimeType`, `webViewLink`, …).

Сохранить в файл:

```bash
node src/cli.js drive corpus-manifest "https://drive.google.com/drive/folders/ВАШ_АРХИВ_2025" > out/corpus-2025-manifest.json
```

**Выгрузка файлов на локальный диск** (тот же обход, что у манифеста: сохраняются относительные пути; Google Docs → `.txt`, Sheets → `.csv`):

```bash
node src/cli.js drive corpus-pull "https://drive.google.com/drive/folders/ВАШ_АРХИВ_2025" "C:\data\corpus-2025-pull" 40 5000
```

Третий и четвёртый аргументы — те же `maxDepth` и `maxFiles` (ограничьте для пробного прогона). В stdout — JSON: `writtenCount`, `failed`, `skipped`, список `written` с парами `path` / `localPath` (при очень большом объёме выводите в файл: `> pull-log.json`).

Пока команда работает, в **консоль (stderr)** печатаются строки `[corpus] …`: сначала обход дерева на Drive, затем прогресс выгрузки. Если «тишина» дольше минуты — чаще всего идёт **скачивание одного большого файла** или медленная сеть; дождитесь сообщения `выгрузка: завершено`.

### Парсить всё подряд (без ручной выборки)

Репозиторий **не встраивает** PDF/OCR для тысяч файлов — это делается в **parserit** (или другом батч-процессе). Чтобы отдать **полный список** файлов архива как очередь:

```bash
node src/cli.js drive corpus-jsonl "https://drive.google.com/drive/folders/ВАШ_АРХИВ" 40 200000 > corpus-all.jsonl
```

В файле **одна строка = один JSON** с полями `driveFileId`, `mimeType`, `path`, `webViewLink`, `documentUrl` (для поля `document_urls` во флоу). Дальше Windmill читает `corpus-all.jsonl` и ставит job на каждую строку. Подробности: [PARSERIT_INTEGRATION.md](PARSERIT_INTEGRATION.md).

---

Инвентаризация структуры **`_lena/tenders`** (другая команда — только дерево Лены, не произвольный архив):

```bash
node src/cli.js drive tenders-inventory "<url_или_id_корневой_папки_Lena>"
```

В stdout — JSON с полями:

- `tendersFolderId` — папка `_lena/tenders`;
- `entries[]` — по каждому тендеру: `year` (или `null` для плоского режима), `tenderFolderName`, `tenderRootFolderId`, число файлов в `inputs`, `drafts`, `exports`, `attachments`, `notes` (только **прямые** дочерние файлы в каждой из этих папок, без рекурсии в подпапки).

Этого достаточно для **обзора масштаба** по `_lena` и для решения, что тащить в контекст или в RAG дальше.

```bash
node src/cli.js drive tenders-inventory "<root>" > out/tenders-inventory.json
```

## Как на основе базы собрать «контекст» для Лены (без векторов)

1. **Ручной курированный слой** — то, что вы уже делаете: выжимки, регламенты, типовые формулировки в `_lena/context` и справочники в `_lena/library`. Их можно пополнять **по шаблону** из лучших прошлых заявок (копипаст с обезличиванием).
2. **Несколько папок на Диске** — переменная **`LENA_EXTRA_CONTEXT_FOLDERS`**: список URL или id папок (через запятую или с новой строки). Файлы из **корня** каждой папки попадают в `drive context-list`, `context-pull` и `agent-bundle` вместе с `_lena/context` (в JSON — поля `lenaContextSource`, `lenaContextExtraRootId`). Каждую папку нужно расшарить на тот же сервисный аккаунт. Подробнее: [GOOGLE_DRIVE.md](GOOGLE_DRIVE.md).
3. **Выборочный экспорт** — для отдельных файлов: `drive download`, для контекста целиком: `drive context-pull` (в локальной копии — подпапки по источнику).
4. **Инвентаризация** — `tenders-inventory` → приоритизировать тендеры с большим `drafts` / `exports` как кандидатов на разбор.

Автоматическое «прочитать все PDF и выписать единый контекст» в этом репозитории **не реализовано** — это зона тяжёлого парсинга (**parserit** / отдельные скрипты) и ручной проверки.

## Векторная база (RAG)

Идея: разбить тексты на **чанки**, посчитать **эмбеддинги**, положить в **векторное хранилище**, на вопрос пользователя — поиск ближайших чанков + ответ LLM.

**Выбранное направление:** векторное хранилище + чанки; пошаговый пайплайн, метаданные чанка, черновой HTTP-контракт и порядок внедрения — в **[RAG_VECTOR_PIPELINE.md](RAG_VECTOR_PIPELINE.md)**. Политика ПДн/коммерческой тайны там пока не раскрыта (отдельный этап).

В `tender-prep` **нет встроенной** векторной БД и вызова эмбеддингов в core CLI: репозиторий даёт **очередь файлов** (`corpus-jsonl`), **выгрузку** (`corpus-pull`) и связку с **parserit**; индекс и запросы — во внешнем сервисе или Windmill.

Пока RAG не подключён к боту, **`/ask` и `/tenderask`** используют только то, что передано в запросе (бандл, список файлов после `/ingest`, текст в команде), а не полнотекстовый корпус на Диске. Локальный прототип индексации и поиска: **`node src/cli.js rag`** — см. [RAG_VECTOR_PIPELINE.md](RAG_VECTOR_PIPELINE.md#rag-cli-mvp).
