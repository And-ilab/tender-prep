# Интеграция с parserit (Windmill)

**parserit** — пространство сценариев в Windmill, где выполняются парсинг и нормализация документов закупки. Сервис **«Лена»** не дублирует этот рантайм: он оркестрирует запросы и потребляет результаты по согласованному контракту.

## Идентификация ресурсов в Windmill

В UI Windmill скрипты и флоу имеют пути вида:

- **Скрипт**: `f/parserit/<имя>` или `u/<user>/<имя>` — в API и MCP часто указывают путь **без** ведущего слэша, как в интерфейсе (например, `u/user/my_script`).
- **Флоу**: папка под `f/`, в описаниях интеграций встречается форма вроде `parserit/it_parser_flow` для флоу, расположенного в UI как `f/parserit/it_parser_flow`.

Конкретные имена скриптов и флоу **не зашиты** в этом репозитории: их фиксирует команда в конфигурации сервиса Лена или в переменных окружения. Ниже — **шаблон** договорённостей; замените плейсхолдеры на актуальные пути из вашего Windmill.

| Плейсхолдер | Пример назначения |
|-------------|-------------------|
| `FLOW_PARSE_TENDER` | Полный разбор комплекта документов в единую структуру |
| `SCRIPT_EXTRACT_LOT` | Быстрое извлечение лота/предмета из извещения |
| `SCRIPT_OCR_TABLE` | Таблицы из сканов (если используется отдельным шагом) |

## Входные данные (типовой контракт)

На стороне parserit ожидается JSON с полями, согласованными с флоу. Типовой набор (адаптируйте под реальный редактор флоу в Windmill):

```json
{
  "tender_id": "внутренний-id-или-реестровый",
  "document_urls": ["https://...", "s3://..."],
  "locale": "ru",
  "options": {
    "extract_requirements": true,
    "extract_deadlines": true
  }
}
```

**Важно:** не передавайте в публичные эндпоинты персональные данные и коммерческую тайну сверх того, что уже разрешено политикой компании и договором с оператором Windmill.

## Полный архив на Drive без ручной выборки

Если нужно **парсить все файлы подряд** в большой папке (архив за год и т.д.), в `tender-prep` нет встроенного движка PDF/OCR — очередь заданий формируется **снаружи** (parserit / свой воркер).

1. Сформируйте **JSONL** — по одной строке JSON на каждый файл на Drive (id, mime, путь, ссылка):

   ```bash
   node src/cli.js drive corpus-jsonl "https://drive.google.com/drive/folders/ARCHIVE_ID" 40 200000 > corpus-all.jsonl
   ```

2. В Windmill (или другом рантайме) **прочитайте файл построчно** и для каждой строки вызовите ваш флоу парсинга: в теле задания используйте поле **`documentUrl`** (или массив `document_urls` из одного элемента) плюс **`path`** как якорь источника в ответе.

3. Либо после **`drive corpus-pull`** обходите **`C:\data\…`** локально тем же parserit, если флоу умеет читать путь на диске вместо URL.

Идемпотентность и кэш по `driveFileId` — на стороне parserit (см. раздел про ошибки ниже).

## Выходные данные

Ожидается структурированный результат (JSON), из которого Лена строит:

- список **требований** с якорями на источник (файл, фрагмент);
- **сроки** и ключевые даты;
- при наличии — **критерии оценки** и штрафные санкции.

Любое поле, которое парсер не смог извлечь надёжно, должно помечаться в ответе (`confidence`, `needs_review`, аналог) — Лена обязана пробрасывать эту пометку в UI или в текст для оператора.

## Вызов из автоматизации

1. Поставить в очередь job Windmill на нужный **script path** или **flow path** с аргументами, совпадающими с полями флоу.
2. При длительном разборе использовать режим ожидания результата или polling по `job_id` — в зависимости от клиента API / MCP.
3. Сохранить сырой ответ parserit (для аудита) отдельно от нормализованного представления в Лене.

## Ошибки и деградация

- **Таймаут** — повторить с идемпотентным `tender_id`, не дублируя загрузку файлов на стороне хранилища, если уже есть кэш.
- **4xx/валидация** — проверить соответствие ключей JSON схеме флоу в Windmill.
- **Частичный результат** — не считать матрицу соответствия полной; явно запросить ручной ввод для пробелов.
