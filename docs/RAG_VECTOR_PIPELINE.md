# Пайплайн RAG: векторная база + чанки

Зафиксированное направление: **извлечённый текст → чанки → эмбеддинги → векторное хранилище → поиск top‑k → подмешивание в промпт LLM**.  
Политика **ПДн и коммерческой тайны** пока **не детализируется** — отдельным этапом позже.

Связка с Диском и очередью файлов: [CORPUS_AND_RAG.md](CORPUS_AND_RAG.md), массовый разбор во Windmill: [PARSERIT_INTEGRATION.md](PARSERIT_INTEGRATION.md).

## 1. Источник «сырья»

| Этап | Команда / артефакт | Результат |
|------|-------------------|-----------|
| Очередь файлов | `drive corpus-jsonl <папка_архива> …` → JSONL | На строку: `driveFileId`, `mimeType`, `path`, `webViewLink`, `documentUrl` |
| Текст | Флоу **parserit** (или другой воркер) по каждой строке: скачать / экспортировать Google Docs → текст; PDF/DOC → извлечение (при необходимости OCR) | Один **нормализованный текст** на файл (или версия «сырой + очищенная») |

Имена папок на Drive могут быть неточными — в метаданные чанка всё равно кладём **`driveFileId`** и ссылку; **смысл поиска** — из текста документа.

## 2. Чанкинг

Рекомендуемые стартовые параметры (подстраиваются по качеству recall):

- **Размер:** ~800–1500 токенов на чанк (или ~3000–6000 символов для русского, если считаете по символам без токенайзера).
- **Перекрытие:** 10–15 % длины чанка, чтобы не резать таблицы и списки пополам без контекста.
- **Границы:** по возможности резать по `\n\n`, заголовкам, нумерации пунктов; для таблиц — не дробить строку посередине, если парсер отдаёт markdown-таблицу.

К каждому чанку добавлять **метаданные** (плоский JSON или поля в БД):

- `drive_file_id`, `web_view_link` (или `document_url`)
- `source_path` — путь в архиве на Drive (как слабый сигнал)
- `chunk_index`, `chunk_total` (опционально)
- `mime_type` исходного файла
- при появлении справочника — `canonical_project_id` / название из CRM (позже)

## 3. Эмбеддинги

- Один выбранный **embedding-модель** на весь индекс (смена модели = переиндексация).
- API в формате, совместимом с OpenAI (`/v1/embeddings`), либо локальная модель — главное, зафиксировать **размерность вектора** в схеме коллекции.

## 4. Векторное хранилище

Выбор за инфраструктурой; критерии:

| Вариант | Плюсы | Заметки |
|---------|--------|---------|
| **Qdrant** | Простой HTTP/gRPC, фильтры по метаданным | Удобно под отдельный микросервис |
| **pgvector** | Всё в Postgres | Если уже есть Postgres |
| **LanceDB** | Файловый/embedded, мало движущихся частей | Удобно для пилота на одной машине |
| Облачный managed RAG | Меньше кода | Зависимость от вендора |

В коллекции хранить: **vector**, **payload** (метаданные + `text` чанка или ссылка на объект в S3/Blob — если тексты огромные).

## 5. Запрос (runtime)

На вопрос пользователя:

1. Эмбеддинг **того же типа**, что при индексации.
2. Поиск **top‑k** (например k=8–20) с опциональным фильтром по `source_path` / году / типу документа.
3. Сборка блока **«Контекст из архива»** для LLM: нумерованные отрывки + обязательно указать, что это выдержки, не выдумывать факты вне них.
4. Вызов основной модели (как сейчас в боте — отдельным шагом после RAG).

Контракт для интеграции с **Telegram-ботом** (когда появится сервис):

```http
POST /v1/rag/query
Content-Type: application/json

{ "query": "текст вопроса", "top_k": 12, "filters": { } }
```

Пример ответа:

```json
{
  "chunks": [
    {
      "text": "…фрагмент…",
      "score": 0.82,
      "metadata": {
        "drive_file_id": "…",
        "web_view_link": "https://…",
        "chunk_index": 3,
        "source_path": "Проект А/…"
      }
    }
  ]
}
```

Бот подставляет `chunks[].text` в пользовательское сообщение перед LLM; ссылки из `metadata` — для сносок в ответе.

## 6. Порядок внедрения (кратко)

1. JSONL по архиву + пилот parserit на **10–50 файлах** разных типов.
2. Скрипт/флоу: **текст → чанки → embeddings → upsert** в выбранную БД.
3. Отдельный **микросервис или Windmill endpoint** с `POST /v1/rag/query`.
4. В `lena-bot.mjs` (или прокси) — вызов RAG перед `chatCompletion` для команд вида `/tenderask` или новой `/askarchive`.

## 7. Реализовано в репозитории: CLI `rag` (MVP)

Локальный пайплайн **без новых npm-пакетов**: `fetch` к OpenAI-compatible **`/v1/embeddings`**, чанки на диске в **`chunks.jsonl`**, поиск — **полный перебор + косинус** в памяти (подойдёт для пилота и средних объёмов; для миллионов чанков позже — Qdrant и т.д.).

```bash
# 1) Выгрузка с Drive (пример) — в папке появятся .txt из Google Docs и бинарники PDF
node src/cli.js drive corpus-pull "<url_архива>" "C:\data\corpus-2025" 30 2000

# 2) Индекс (нужен ключ и совместимый POST …/embeddings — см. ниже про DeepSeek)
set LENA_EMBEDDING_MODEL=text-embedding-3-small
node src/cli.js rag index "C:\data\corpus-2025" "C:\data\rag-index-2025"

# 3) Поиск
node src/cli.js rag query "C:\data\rag-index-2025" "как оформить отзыв заказчика" 8

# 4) Поиск + ответ LLM (тот же ключ; чат — LENA_OPENAI_MODEL)
node src/cli.js rag ask "C:\data\rag-index-2025" "какие были требования к срокам?" 10
```

Индексируются только **текстовые** расширения: `.txt`, `.md`, `.csv`, `.log`. Папки `.git`, `node_modules`, `.lena-rag` пропускаются.

Переменные: **`LENA_OPENAI_API_KEY`** или **`OPENAI_API_KEY`**, **`LENA_OPENAI_BASE_URL`**, **`LENA_OPENAI_MODEL`** (чат в `rag ask` и в боте), **`LENA_EMBEDDING_MODEL`**, опционально **`LENA_EMBEDDING_BASE_URL`** (отдельный базовый URL для `POST …/embeddings`), опционально **`LENA_EMBEDDING_API_KEY`** (если эмбеддинги у другого провайдера, чем чат; иначе используется тот же ключ, что для чата), опционально **`LENA_EMBEDDING_DIMENSIONS`**.

**DeepSeek и RAG:** для чата можно указать `LENA_OPENAI_BASE_URL=https://api.deepseek.com/v1` и модель DeepSeek в **`LENA_OPENAI_MODEL`**. У DeepSeek в OpenAI-совместимом API **нет** рабочего **`POST /v1/embeddings`** для текущего клиента: индекс **`rag index`** должен ходить на хост с **`/embeddings`** — облачный API, **или локальный CPU-сервер** из [scripts/local_openai_embeddings/README.md](../scripts/local_openai_embeddings/README.md) (`LENA_EMBEDDING_BASE_URL=http://127.0.0.1:8765/v1`).

Аргументы **`rag index`**: `[maxChars] [overlapChars] [maxFiles]` — числа в конце строки команды.

Интеграция с **Telegram-ботом** (`/askarchive`) в коде бота пока **не подключена** — используйте `rag ask` из терминала или вынесите вызов в свой сервис по HTTP-контракту из раздела 5.

**Снятие текста с .docx / .pdf / .doc** с выгрузки `corpus-pull`: отдельный Python-скрипт (без удаления оригиналов) — [scripts/corpus_extract_text/README.md](../scripts/corpus_extract_text/README.md); затем `rag index` по папке с `.txt`.

## 8. Внешняя векторная БД (следующий шаг)

Для очень больших корпусов по-прежнему логичны **Qdrant / pgvector / managed RAG**: тот же чанкинг и тот же API эмбеддингов, но вместо полного скана — upsert в БД и ANN-поиск. Репозиторий не подключает эти SDK намеренно.
