# PRODUCT_CONTEXT · сервис «Лена»

Документ задаёт **единый продуктовый контекст** для разработки, сопровождения и для агентов (в т.ч. в Cursor): что такое Лена, какие цели у tender-prep и что считается вне зоны ответственности.

## Проблема

Участник закупок тратит много времени на чтение положений, технических заданий и форм: нужно вытащить обязательные требования, сроки, критерии оценки и сопоставить их с возможностями поставщика без пропусков и противоречий.

## Решение

**«Лена»** — специалист по подготовке тендерных документов (не «универсальный помощник» в чате):

- **сама формирует пакет материалов**: структурирует требования из пакета документов закупки, ведёт матрицу соответствия (требование → доказательство/раздел заявки), готовит тексты и черновики под формат заказчика, с явными пометками, где нужна проверка юриста или эксперта;
- в общем чате с командой **по необходимости** задаёт **узкие уточняющие вопросы**; основная работа — выдача готовых фрагментов и структуры заявки, а не сопровождение в режиме консультанта.

**tender-prep** — репозиторий, где фиксируются:

- продуктовые допущения и глоссарий;
- договорённости по интеграции с **parserit** (см. [PARSERIT_INTEGRATION.md](PARSERIT_INTEGRATION.md));
- инструкции для среды разработки, в частности **Git на Windows** ([WINDOWS_GIT.md](WINDOWS_GIT.md)).

## Пользователи и сценарии

| Роль | Типовой сценарий |
|------|------------------|
| Менеджер по тендерам | Загрузка комплекта, запуск парсинга, проверка матрицы, выгрузка в шаблон заявки |
| Технический специалист | Настройка связки с Windmill, обновление путей к флоу, контроль версий промптов/правил |
| ИИ-агент в IDE | Чтение этого файла и смежных `docs/*.md` перед правками кода или генерацией ответов по тендеру |

## Базовый сценарий интеграции (цель продукта)

Зафиксированные допущения на этапе разработки:

1. **Площадка** — в приоритете и в объёме первых версий только **IceTrade** (другие ЭТП — позже, отдельным контуром).
2. **Хранилище артефактов** — **только Google Drive** по согласованной структуре `_lena/tenders/…` (карточка, документы закупки, черновики, экспорт и т.д.). Локальные копии — вспомогательный инструмент разработчика/воркера, не источник истины для команды.
3. **Точка входа в чате (целевое поведение)** — упоминание Лены и **URL карточки IceTrade** → дальше по целевому пайплайну ниже; при неясностях — опора на архивный RAG; если недостаточно — **вопрос в чат человеку** (без домыслов).

### Целевой пайплайн после ссылки на тендер (IceTrade → Drive → менеджер)

Фиксируем **ожидаемую цепочку** (реализация в коде/воркерах наращивается по шагам):

1. **Идентификация** — из URL карточки получить id закупки на площадке и сопоставить с **`tender_id`** папки на Drive (`_lena/tenders/…/<tender_id>/`), создав дерево при необходимости.
2. **Загрузка комплекта** — скачать файлы документации закупки: на карточке **IceTrade** встречаются и **прямые вложения с площадки**, и **ссылки на файлы с goszakupki.by** (единый каталог/выдача); bootstrap должен обрабатывать **оба варианта** в одном прогоне.
3. **Размещение на Drive** — сохранить сырой комплект в **`inputs/`** этой закупки. В продуктовых текстах эту папку называем **«документы заказчика»** (комплект извещения и приложений заказчика для данного тендера); техническое имя в репозитории остаётся `inputs` (см. [GOOGLE_DRIVE.md](GOOGLE_DRIVE.md)).
4. **Распознавание и разбор** — извлечение текста/таблиц (OCR при необходимости), нормализация через согласованный контур **parserit** / Windmill ([PARSERIT_INTEGRATION.md](PARSERIT_INTEGRATION.md)); результат — структурированные требования, сроки, якоря на файлы.
5. **Анализ «что закрыто автоматически / что нет»** — по тексту закупки и доступным источникам (в т.ч. `_lena/org-docs`, `_lena/founding-docs`, уже загруженные файлы тендера) определить, **что Лена не может подготовить или достать самостоятельно** (нет первичного документа, нужен оригинал с подписью/печатью, нужны данные только у менеджера, внешний сертификат без доступа и т.п.).
6. **Запрос менеджеру** — сформировать **явный перечень недостающих документов и данных** и запросить их у менеджера в рабочем чате; операционно закрепить переписку в **`notes/telegram-managers-log.md`** на этом тендере ([LENA_RULES.md](LENA_RULES.md) §6e). До появления недостающих материалов не притворяться, что комплект полон ([LENA_RULES.md](LENA_RULES.md) §6f, §6g).

Первый автоматический шаг (**bootstrap**) по ссылке IceTrade в **боте** и через **`node src/cli.js tenders icetrade-bootstrap`** уже есть: дерево тендера на Drive, попытка скачать вложения в **inputs** (в том числе смешанный комплект **IceTrade + goszakupki.by**), заметка в **notes** с чеклистом запросов менеджеру. Полноценный **разбор текста** (parserit / OCR) подключается отдельно поверх загруженных файлов.

### Корпус архива и RAG — только после решения об участии

Материалы закупки **не добавляются** в долговременный текстовый корпус / векторный индекс автоматически при первичном разборе.

- Имеет смысл пополнять архив **после** того, как **сформирован и утверждён пакет документов на участие** по данному тендеру (или после явного решения «идём», когда пакет готов к финальной правке).
- Добавление в корпус для последующего «обучения» системы (накопление знаний, не fine-tuning весов) выполняется **только по явному действию человека** (команда, кнопка, отдельный флоу): участие могут **не планировать**, данные закупки не должны попадать в общий RAG без согласия.

Реализация триггера и проверок consent — отдельная задача в коде/боте; до её появления оператор отражает политику вручную (не выполнять `corpus-pull`/индексацию по закупке без решения).

## Ключевые сущности

- **Закупка** — логическая единица: извещение, документация, формы, возможные разъяснения.
- **Лот / предмет** — то, на что подаётся заявка в рамках закупки.
- **Требование** — формулировка из документации, которую нужно закрыть документом или текстом в заявке.
- **Матрица соответствия** — таблица «требование → статус → ссылка на приложение/раздел».
- **Результат parserit** — структурированный вывод пайплайна Windmill (JSON/таблица), который Лена или оператор используют как основу, а не как юридически заверенный факт.

## Границы ответственности

**В зоне Лены и tender-prep**

- Подготовка структуры и черновиков под конкретную закупку.
- Согласованные вызовы parserit и обработка ожидаемых форматов ответа.

**Вне зоны (явная эскалация человеку)**

- Юридическая квалификация рисков, подписание заявки, решение об участии.
- Окончательная проверка соответствия законодательству и регламенту площадки.
- Любые действия от имени организации на ЭТП без явного поручения.

## Принципы для агентов

Развёрнутые операционные правила (матрица, черновики, файлы вне текста, промпты) — в [LENA_RULES.md](LENA_RULES.md). Кратко:

1. Не выдумывать требования: опираться на текст закупки или на помеченный вывод parserit.
2. Помечать неуверенность и ссылаться на источник (файл, страница, пункт).
3. Соблюдать контракт интеграции с parserit: имена полей, пути флоу, обработка ошибок и таймаутов.
4. При работе в Windows учитывать особенности Git из [WINDOWS_GIT.md](WINDOWS_GIT.md), чтобы не портить диффы и кодировки.
